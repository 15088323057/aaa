# 《动态页面加载功能文件包含漏洞分析报告》

## 摘要

本次安全测试针对用户管理系统（Python Flask Web 应用）中新增加的动态页面加载功能开展。测试发现，`/page` 路由未对用户输入的页面名称参数进行任何合法性校验，攻击者可通过在 `name` 参数中注入 `../` 路径遍历序列，突破 `pages/` 目录限制，将服务器上任意文件作为页面内容包含（Include）到首页模板中渲染并返回给客户端。在验证过程中，成功实现了对系统级敏感文件 `/etc/passwd`、`/etc/shadow`（含密码哈希值）以及应用源代码 `app.py` 的包含读取。由于模板渲染使用 `{{ page_content | safe }}` 过滤器，包含的文件内容中的 HTML 和 JavaScript 脚本将在用户浏览器中直接解析执行，形成持久性跨站脚本攻击（Stored XSS）的利用条件。该漏洞攻击复杂度低、无需身份认证、无需用户交互，可稳定远程触发。根据 CVSS 3.1 标准，风险等级评定为 **高危（High）**。核心修复方案为使用 `os.path.realpath()` 对路径进行规范化处理，并校验最终路径是否位于预期目录范围内。

## 1. 项目背景与测试目标

本项目为一个基于 Python Flask 框架构建的用户信息管理平台，主要用于演示 Web 安全漏洞及其修复方法。平台已实现用户注册、登录、个人信息管理、头像上传、账户充值、用户搜索及安全报告下载等功能。

本次新增的动态页面加载功能旨在允许通过 URL 参数动态加载并显示 `pages/` 目录下的 HTML 页面内容。由于采用服务端文件包含（Server-Side File Inclusion）的设计模式——即根据用户提供的参数拼接路径、读取文件、并将内容嵌入到服务端模板中渲染输出——该功能天然存在文件包含漏洞的风险。

本次测试的目标为：

- 验证 `/page` 路由是否存在文件包含漏洞（Local File Inclusion, LFI）；
- 确认攻击者能够通过该漏洞包含哪些类型的文件及其可读范围；
- 评估 `{{ page_content | safe }}` 渲染方式对漏洞危害的放大效应；
- 评估漏洞的实际危害等级；
- 提出可执行的修复方案并进行修复验证。

本次测试在授权的实验环境中进行，遵循最小影响原则，未进行任何数据破坏、服务中断或未授权修改操作。

## 4. 漏洞概述

文件包含漏洞（File Inclusion Vulnerability）是指应用程序在将用户输入作为文件路径参数传递给文件包含函数（如 PHP 的 `include()`、`require()`，或 Python 中通过模板引擎渲染外部文件内容）时，未对输入进行严格校验，导致攻击者能够控制被包含的文件。

在本系统中，`/page` 路由采用服务端文件包含的设计模式：

1. 接收用户通过 URL 参数 `name` 提供的页面名称；
2. 拼接路径为 `"pages/" + name`；
3. 读取对应文件内容；
4. 将文件内容通过 `{{ page_content | safe }}` 嵌入到首页模板中渲染输出。

这一设计存在两个层面的安全缺陷：

- **文件包含控制层面**：攻击者可通过 `../` 序列控制被包含的文件路径，包含任意系统文件。
- **模板渲染层面**：`| safe` 过滤器标记内容为安全的 HTML，Jinja2 模板引擎不会对其进行 HTML 实体转义。这意味着如果被包含的文件中包含 HTML 或 JavaScript 代码，这些代码将在用户浏览器中直接解析执行。

攻击者无需身份验证即可远程利用该漏洞，利用难度极低。

## 5. 漏洞原理分析

### 5.1 正常业务逻辑

动态页面加载功能的设计意图为：用户通过 URL 参数指定一个页面名称（如 `help`），系统在 `pages/` 目录下查找对应的 HTML 文件，读取其内容，并将内容渲染到首页模板中展示给用户。

### 5.2 实际处理方式

后端代码在处理文件包含时，采用了以下流程：

```
用户输入（name 参数）
  → os.path.join("pages", name)        # 路径拼接
  → os.path.exists() / os.path.isfile() # 文件存在性检查
  → open(path, "r").read()              # 文件内容读取
  → render_template("index.html", page_content=内容)  # 模板渲染
  → {{ page_content | safe }}           # 标记为安全HTML，不转义输出
```

### 5.3 安全控制缺失位置

本漏洞包含两个层面的安全缺陷：

**第一层：文件包含路径控制缺失**

```python
page_path = os.path.join("pages", name)
```

此处直接使用用户输入的 `name` 参数与 `"pages"` 进行路径拼接，未对 `name` 中是否包含路径遍历字符（如 `../`）进行任何校验。`os.path.join()` 在遇到以 `../` 开头的相对路径时，会正常处理目录回退逻辑。因此，当 `name = "../../etc/passwd"` 时，拼接结果为 `pages/../../etc/passwd`，经过操作系统路径解析后等同于 `/etc/passwd`，成功实现了跨目录文件包含。

**第二层：模板输出编码缺失**

```html
{{ page_content | safe }}
```

Jinja2 模板引擎默认会对所有变量进行 HTML 实体转义（如 `<` → `&lt;`），以防止 XSS 攻击。但 `| safe` 过滤器会标记该内容为"安全的"，告诉模板引擎不需要转义。当被包含的文件（如攻击者上传的包含恶意代码的文件，或攻击者通过其他方式写入的文件）包含 `<script>` 标签或 HTML 事件处理器时，这些代码将不经转义地直接输出到浏览器中解析执行。

### 5.4 文件包含的数据流

```
攻击者构造请求：
GET /page?name=../../../../etc/passwd

数据流向：
① URL 参数 name = "../../../../etc/passwd"
② os.path.join("pages", "../../../../etc/passwd")
   → "pages/../../../../etc/passwd"
   → 操作系统解析为 "/etc/passwd"
③ os.path.exists("/etc/passwd") → True
④ open("/etc/passwd", "r").read() → 读取文件全部内容
⑤ render_template("index.html", page_content=文件内容)
⑥ 模板输出：{{ page_content | safe }}
   → 文件内容原样嵌入 HTML 返回给客户端
⑦ 客户端（浏览器）收到包含 /etc/passwd 内容的页面
```

### 5.5 XSS 攻击链（扩展威胁）

如果攻击者能够通过其他方式（如头像上传功能、数据库注入等）在服务器上写入一个包含恶意 JavaScript 的文件，则可通过文件包含漏洞引入该文件：

```
① 攻击者上传包含 <script>alert(document.cookie)</script> 的文件
② 文件保存到 static/uploads/avatar.html
③ 攻击者请求 /page?name=../static/uploads/avatar.html
④ 文件内容被包含到首页模板中
⑤ JavaScript 在访问首页的用户浏览器中执行
⑥ 形成持久性跨站脚本攻击（Stored XSS）
```

### 5.6 根本原因分类

本漏洞的根本原因分为两个层次：

1. **文件包含层面的直接原因**：输入校验缺失（Missing Input Validation）——未对用户输入的路径参数进行规范化处理和目录边界校验。
2. **输出编码层面的直接原因**：安全标记滥用——不应将外部文件内容标记为 `| safe` 输出，外部可控内容未经转义直接渲染。

## 6. 漏洞发现过程

该漏洞是代码实施后通过白盒代码审计（Code Review）发现的。以下为针对性的验证步骤：

**步骤 1：** 浏览 `/page` 路由的代码实现，发现路径拼接使用了 `os.path.join("pages", name)` 且未对 `name` 进行任何过滤或校验。

**步骤 2：** 检查模板文件 `index.html`，发现 `page_content` 变量使用 `{{ page_content | safe }}` 渲染，确认外部可控内容被标记为安全 HTML 输出。

**步骤 3：** 发送正常请求验证功能可用性：

```
GET /page?name=help
```

返回 HTTP 200，页面内容区域显示帮助中心页面内容。正常功能正常运作。

**步骤 4：** 构造文件包含测试请求，试图包含预期目录之外的文件：

```
GET /page?name=../app.py
```

页面成功显示了 `app.py` 的源代码内容，确认服务端文件包含可跨目录执行。

**步骤 5：** 扩大包含范围，尝试包含系统级敏感文件：

```
GET /page?name=../../../../etc/passwd
```

页面成功返回 `/etc/passwd` 文件内容，确认文件包含范围不受目录深度限制。

**步骤 6：** 继续尝试包含更敏感的系统文件 `/etc/shadow`：

```
GET /page?name=../../../../etc/shadow
```

页面成功返回 `/etc/shadow` 文件内容，包含系统中所有用户的密码哈希值，确认文件包含可触及最敏感的系统文件。

**步骤 7：** 验证 `| safe` 过滤器的效果，构造包含 HTML 标签的测试：

```
GET /page?name=help
```

观察响应中帮助中心的 HTML 标签被正确渲染，未进行 HTML 实体转义，确认 `| safe` 过滤器生效。

### 7.4 漏洞成立依据

1. **攻击者可控制被包含的文件路径**：用户输入的 `name` 参数直接决定服务器读取哪个文件，路径中的 `../` 序列被操作系统正常解析。
2. **跨目录包含成������**：`app.py`、`/etc/passwd`、`/etc/shadow` 等 `pages/` 目录之外的文件均被成功包含到服务端模板中输出。
3. **文件包含范围不受限**：从项目源代码到系统级敏感文件均可成功包含。
4. **`| safe` 过滤器扩大了危害面**：被包含文件中的 HTML 标记被浏览器正常解析，若包含恶意脚本将在用户浏览器中执行。
5. **无需身份验证**：所有测试均在未登录状态下完成。

## 8. 漏洞影响分析

### 8.1 机密性影响

**严重。** 本漏洞直接导致服务器文件系统上任意可读文件的内容被包含到服务端输出中泄露给客户端，具体包括：

- **应用源代码**：`app.py` 等核心代码泄露，导致攻击者能够分析认证逻辑、密钥管理和所有业务实现细节，为后续攻击提供信息支撑。
- **系统配置文件**：`.gitignore`、配置文件的泄露可能暴露项目结构、第三方服务连接信息和部署配置。
- **系统敏感文件**：`/etc/passwd` 和 `/etc/shadow` 的泄露直接暴露了系统用户列表和密码哈希值。攻击者可使用 Hashcat、John the Ripper 等工具尝试离线破解弱口令，一旦成功即可获得系统级访问权限。
- **数据库文件**：通过源代码分析获得数据库路径后，可进一步包含 SQLite 数据库文件 `data/users.db`，获取所有用户的密码哈希和个人信息。
- **环境变量和密钥**：若运行环境将敏感信息存储在环境变量文件或配置文件中，均可通过文件包含获取。

### 8.2 完整性影响

**间接影响。** 当前漏洞直接用于文件读取（包含到输出），不涉及对服务端文件的新增、修改或删除。然而：

- 通过泄露的 `FLASK_SECRET_KEY`（存储在源代码或环境变量中），攻击者可以伪造合法 session，冒充任意用户身份执行操作，从而间接影响数据完整性。
- 通过包含恶意 HTML 文件形成 XSS 攻击链，可在用户浏览器中执行任意 JavaScript，修改页面内容或发起 API 请求。

### 8.3 可用性影响

**不直接受影响。** 本漏洞不会直接导致服务中断、程序崩溃或拒绝服务。

### 8.4 业务影响

本漏洞可能对实际业务产生以下影响：

- **凭据泄露**：应用密钥（`FLASK_SECRET_KEY`）如果被包含获取，攻击者可以伪造合法 session，冒充任意用户身份登录系统。
- **供应链安全风险**：源代码完全泄露使得攻击者可以挖掘更多 0day 漏洞，或直接复制整个应用的技术方案。
- **合规与法律风险**：如果系统包含用户个人数据（姓名、电话、邮箱、地址等），批量泄露可能导致违反《个人信息保护法》等法律法规，引发监管处罚和声誉损失。
- **权限提升链**：通过 `/etc/shadow` 获取的密码哈希值可能被破解，攻击者有机会从 Web 应用层面的文件包含漏洞利用升级为服务器操作系统权限获取。
- **跨站脚本攻击链**：文件包含与头像上传功能配合，攻击者可上传包含恶意 JavaScript 的 HTML 文件，然后通过文件包含引入该文件，形成针对管理员的 Stored XSS 攻击，进一步窃取管理员 session。

### 8.5 影响边界

**已验证范围：** 可包含 Web 应用工作目录和系统 `etc/` 目录下的任意可读文件到服务端响应中。`/etc/shadow` 在本实验环境中可读是由于容器内进程以 root 权限运行。

**尚未验证：** 未验证完整的 XSS 攻击链（上传恶意 HTML + 文件包含触发）；未验证 Windows 环境下的文件包含兼容性；未验证是否可以利用文件包含执行服务器端脚本（在 PHP 环境下 LFI 通常可导致 RCE，在 Flask 环境下需结合其他条件）。

## 10. 漏洞根因分析

### 10.1 直接技术原因

本漏洞包含两个层面的直接技术原因：

**第一层：文件包含路径控制缺失**

1. **缺少输入校验**：未对 `name` 参数中可能包含的路径遍历序列（`../`、`..\\` 等）进行拦截或过滤。
2. **缺少路径边界约束**：使用 `os.path.join("pages", name)` 后未使用 `os.path.realpath()` 或 `os.path.abspath()` 规范化路径，且未校验最终路径是否以期望的 `pages/` 目录开头。
3. **文件包含函数使用不当**：`open(path, "r").read()` 作为一个广义的"文件包含"操作，未对包含范围做任何限制。

**第二层：模板输出编码缺失**

4. **安全标记滥用**：使用 `| safe` 过滤器将外部文件内容标记为安全 HTML，Jinja2 模板引擎不会进行 HTML 实体转义。外部文件内容可能是不可信的，不应标记为安全。
5. **缺少内容类型校验**：未验证被包含文件的 MIME 类型或内容格式，纯文本文件（如 `.py`、`.gitignore`）的内容也被直接嵌入 HTML 输出。

### 10.2 开发流程原因

1. **安全编码规范不完善**：未建立针对文件包含/路径操作的安��编码标准，开发者未意识到 `| safe` 过滤器在外部内容场景下的安全风险。
2. **代码审查缺失**：该功能在开发完成后未经过安全代码审查（Security Code Review），文件包含和输出编码两个层面的缺陷均未被发现。
3. **测试用例不全面**：功能测试仅覆盖了正常输入场景，未包含路径遍历、HTML 注入等边界测试和负面测试用例。

### 10.3 管理与防护原因

1. **缺少安全基线配置**：未使用 WAF 或 RASP 等运行时保护措施作为纵深防御。
2. **日志告警不足**：异常文件访问模式未纳入安全事件监控范围。
3. **运行权限管控不足**：Web 应用进程权限未按照最小权限原则配置，导致可包含读取系统级敏感文件。

## 11. 修复方案

### 11.1 紧急处置措施

在完成永久修复前，建议立即执行以下措施以降低风险：

1. **临时禁用 `/page` 路由**：在 `app.py` 中注释或删除 `/page` 路由定义，阻止所有对该功能的访问。
2. **移除 `| safe` 过滤器**：从模板文件中移除 `| safe`，让 Jinja2 对内容进行默认转义，消除 HTML 注入风险。
3. **限制接口来源**：通过反向代理配置 IP 白名单，限制 `/page` 路由的可访问来源。
4. **排查访问日志**：检查服务器访问日志中是否存在异常的 `/page` 请求（包含 `../` 或 `%2f` 等序列），确认是否已被利用。
5. **更换可能泄露的凭据**：如果 `FLASK_SECRET_KEY` 被泄露，应立即更换并使所有现有 session 失效。

**重要：** 紧急处置措施仅为临时缓解手段，不能替代根本修复。

### 11.2 根本修复方案

**修复策略一：路径规范化校验（防止跨目录文件包含）**

对用户输入的路径参数进行规范化处理，并校验最终路径是否位于预期的 `pages/` 目录范围内。

**修复代码（Python Flask）：**

```python
import os

@app.route("/page")
def dynamic_page():
    name = request.args.get("name", "")
    page_content = "页面不存在"

    if name:
        # 修复1：使用 os.path.realpath() 规范化路径
        pages_dir = os.path.join(BASE_DIR, "pages")
        safe_path = os.path.realpath(os.path.join(pages_dir, name))

        # 修复2：验证路径是否仍在 pages 目录内
        if not safe_path.startswith(os.path.realpath(pages_dir) + os.sep):
            # 路径越界，拒绝包含
            page_content = "页面不存在"
        elif not os.path.exists(safe_path) or not os.path.isfile(safe_path):
            # 尝试加 .html 后缀
            safe_path_html = safe_path + ".html"
            if os.path.exists(safe_path_html) and os.path.isfile(safe_path_html):
                try:
                    with open(safe_path_html, "r", encoding="utf-8") as f:
                        page_content = f.read()
                except Exception:
                    page_content = "页面读取失败"
            else:
                page_content = "页面不存在"
        else:
            try:
                with open(safe_path, "r", encoding="utf-8") as f:
                    page_content = f.read()
            except Exception:
                page_content = "页面读取失败"

    username = session.get("username")
    user_info = get_user_info(username)
    return render_template("index.html", user=user_info, search_results=None, keyword="", page_content=page_content)
```

**修复原理：**

- `os.path.realpath()` 会解析路径中所有的符号链接和 `..`/`.` 序列，返回文件系统上的真实绝对路径。例如 `pages/../../etc/passwd` 会被解析为 `/etc/passwd`。
- `safe_path.startswith(os.path.realpath(pages_dir) + os.sep)` 校验解析后的路径是否仍然以 `pages/` 目录的真实路径开头。如果 `../` 导致路径越出 `pages/` 目录，则此校验会失败，拒绝文件包含。
- 该方式既支持 `pages/` 目录下的子目录结构，又杜绝了所有形式的 `../` 遍历攻击。

**修复策略二：移除或谨慎使用 `| safe` 过滤器**

```html
<!-- 修复前：外部内容被标记为安全，不转义输出 -->
{{ page_content | safe }}

<!-- 修复后：让 Jinja2 进行默认的 HTML 实体转义 -->
{{ page_content }}
```

如果确实需要渲染合法 HTML（如帮助中心页面包含格式化标签）：
- 方案A：使用白名单 HTML 清理库（如 `bleach`）过滤后输出，只允许安全的标签和属性。
- 方案B：将帮助页面内容的渲染方式改为仅在确认内容来源可信时才使用 `| safe`。
- 方案C：改用模板继承机制，将不同页面作为独立模板，而非从文件系统动态读取并嵌入。

**为什么路径白名单（方案C）是更安全的做法：**

```python
# 更安全的替代方案：白名单方式
ALLOWED_PAGES = {"help", "about", "faq", "help.html", "about.html", "faq.html"}

@app.route("/page")
def dynamic_page():
    name = request.args.get("name", "")
    
    # 白名单校验：仅允许预定义的页面名称
    if name not in ALLOWED_PAGES:
        page_content = "页面不存在"
    else:
        page_path = os.path.join(BASE_DIR, "pages", name)
        with open(page_path, "r", encoding="utf-8") as f:
            page_content = f.read()
    ...
```

白名单方式从根本上消除了路径拼接的歧义，是最安全的文件包含控制方式。但如果页面数量动态增长且无法预定义，则应使用路径规范化校验方式。

### 11.3 纵深防御措施

1. **最小权限原则**：Web 应用进程不应以 root 权限运行，应使用专门的低权限系统用户。限制 Web 进程对文件系统的读取范围。
2. **路径白名单**：如果动态页面列表是可枚举的，使用白名单方式直接限定可加载的页面名称列表。
3. **内容安全策略（CSP）**：设置 `Content-Security-Policy` 响应头，限制页面中可执行的脚本来源，降低 XSS 利用的影响。
4. **安全响应头**：增加 `X-Content-Type-Options: nosniff` 头，防止浏览器进行 MIME 类型嗅探。
5. **输入验证**：对所有用户输入进行服务端参数校验，建议对 `name` 参数仅允许字母、数字、连字符和下划线。
6. **输出编码**：对外部文件内容进行 HTML 实体转义输出，仅在确认内容安全时使用白名单过滤方式渲染。
7. **日志审计**：记录所有文件包含操作的来源 IP 和被包含文件名，对异常路径模式（含 `../` 等）进行告警。

### 11.4 不推荐的修复方式

| 不推荐的方法 | 原因 |
| ---- | ---- |
| 仅替换 `../` 为空字符 | 可被 `....//`（嵌套遍历）、`..%2f`（URL 编码）、`..%252f`（双重编码）等方式绕过 |
| 仅依赖前端 JavaScript 校验 | 攻击者可直接使用 curl、Burp Suite 等工具绕过前端校验 |
| 仅使用黑名单过滤关键字 | 黑名单难以覆盖所有编码和变体形式 |
| 仅移除 `\| safe` 而不修复路径控制 | XSS 风险消除但文件包含读取任意文件的问题仍然存在 |
| 仅隐藏接口地址 | 接口地址可通过源代码分析、网络抓包等方式获取 |
| 仅部署 WAF 而不修复代码 | WAF 规则可被绕过，且无法处理业务逻辑层面的路径控制问题 |

## 13. 安全加固建议

1. **建立安全编码规范**：制定包含文件路径操作、输出编码、权限校验等场景的安全编码规范，特别强调"永远不要信任用户输入"的基本原则。

2. **模板安全培训**：对开发团队进行 Jinja2/Flask 模板安全专项培训，重点讲解 `| safe` 过滤器、`autoescape` 开关和模板注入的风险场景。

3. **集成自动化安全测试**：在 CI/CD 流水线中集成 SAST（静态应用安全测试）工具（如 Bandit、Semgrep），自动检测文件包含、路径遍历、模板注入等常见漏洞模式。

4. **实施代码安全审查制度**：对涉及文件操作、模板渲染、权限校验的代码变更，强制要求经过安全专职人员的代码审查。

5. **运行时自我保护**：在容器化部署中使用 seccomp、AppArmor 或 SELinux 策略限制 Web 进程可访问的文件路径范围。

6. **安全基线与配置管理**：制定服务器安全基线，包括：Web 进程不以 root 运行、敏感文件权限严格限制、日志定期轮转归档。

7. **多层次输出编码策略**：在模板层、应用层和 Web 服务器层建立多层次的输出编码策略，防止单点安全失效。

## 14. 实验局限性

1. **未获得完整源代码**：本实验基于当前工作目录中的代码进行分析，对 Flask/Jinja2 框架底层行为的理解基于公开文档而非框架源码级别的审计。
2. **测试环境限制**：测试在容器化开发环境中进行，未在生产环境验证漏洞的可利用性和影响范围。
3. **未进行破坏性验证**：遵循最小影响原则，未尝试通过文件包含获取的数据进行后续提权攻击验证。
4. **未完成 XSS 完整攻击链验证**：理论分析了文件包含 + `| safe` 过滤器 + 头像上传的 Stored XSS 攻击链，但未完成端到端的完整验证。
5. **风险评分基于有限条件**：CVSS 评分基于当前已知条件进行评估，实际生产环境中的风险等级可能因数据敏感度、网络架构、运行权限等因素而有所不同。
6. **未测试 PHP 风格 LFI 的 RCE 效果**：在 PHP 环境下，通过包含攻击者控制的日志文件等方式可升级为远程代码执行（RCE）。在 Python/Flask 环境下，此升级路径需要结合其他条件（如写文件能力），未在本次测试中深入验证。

## 15. 结论

本次安全测试确认 `/page` 路由存在本地文件包含漏洞（LFI），核心证据为通过注入 `../` 序列成功将系统敏感文件 `/etc/shadow`（含密码哈希值）和应用源代码 `app.py` 包含到服务端输出中。该漏洞的根本原因是两个安全缺陷的组合：

1. **文件包含路径控制缺失**：未使用 `os.path.realpath()` 规范化路径，未校验被包含文件是否位于预期目录范围内；
2. **输出编码缺失**：使用 `| safe` 过滤器将外部文件内容标记为安全 HTML，扩大了攻击面。

该漏洞风险等级评定为 **高危**，攻击者无需身份认证即可远程稳定触发，利用难度极低。漏洞直接影响系统机密性（任意文件读取），并可间接影响数据完整性（通过泄露的密钥伪造 session）和用户安全性（通过 `| safe` 形成 XSS 攻击链）。

**最优先的修复动作**为：（1）实施路径规范化校验（使用 `os.path.realpath()` + 目录前缀检查）；（2）重新评估 `| safe` 过滤器的使用安全性。修复完成后应执行完整的回归测试和针对路径遍历、HTML 注入、编码绕过的安全复测。本次测试中的修复已通过验证。
